# Agent 核心概念与架构

上一篇讲了 Agent 是什么，这一篇我想带你把引擎盖掀开，看看它内部到底是怎么转的。别担心，我不会堆术语，只讲那几个真正重要的概念。把这些理解透了，你再去读任何一个 Agent 产品的文档，都会有种"哦，原来说的是这个"的秒懂感。

## 核心引擎：那个"想—做—看"的循环

几乎所有现代 Agent 的骨架，都是一个不断重复的循环，业界给它起了个名字叫 ReAct，也就是 Reason 加 Act，推理加行动。说白了就是三步：先想一下"我下一步该干啥"，然后动手调用某个工具去做，接着看看工具返回了什么结果，再根据结果决定下一步——没完成就继续绕，完成了就把结果交给你。

光这么说有点抽象，我给你还原一段真实的循环长什么样。假设你让它修一个 bug，它的内心戏大概是这样的：先想"要修 bug 我得先找到相关代码"，于是去 grep 一下 login 关键字，看到命中了 auth.js 第 42 行；接着想"看看这个文件"，读进来发现问题出在第 42 行的空判断，于是动手改掉；改完它不会就此收工，而是想"得跑测试验证一下"，运行测试，看到全部通过，这才安心地回过头告诉你"修好了"。

这里有个我特别想强调的点：**一个 Agent 强不强，一半看模型，另一半就看这个循环里能拿到多好的反馈。** 测试报错、编译错误、命令输出，这些客观的反馈就是 Agent 自我纠错的依据。反馈越扎实，它就越能像个靠谱的人一样自己把活兜住；反馈一旦模糊，它就容易越跑越偏。

## 工具调用：Agent 的手和眼

工具是 Agent 用来接触世界的手和眼。现代大模型被专门训练成能输出结构化的意图——"我要调用工具 X，参数是 Y"——外层程序执行完，再把结果喂回给它。常见的工具无非几类：读写文件、搜索这样的文件类操作，跑 shell、执行代码、跑测试这样的执行类操作，联网搜索、抓网页、查数据库这样的信息类操作，以及发邮件、调第三方 API、操作浏览器这样直接对外产生动作的操作。

对一个编程 Agent 来说，好不好用很大程度上就取决于它的文件和执行工具扎不扎实——搜索准不准、编辑会不会改错地方、命令沙箱安不安全。这些看起来不起眼的细节，恰恰是拉开产品体验差距的地方。

## MCP：给 Agent 插工具的"USB 接口"

早期每个 Agent 都自己写一套工具集成，彼此互不兼容，重复造轮子。Anthropic 在 2024 年底提出的 MCP，也就是 Model Context Protocol，很快就成了跨产品的事实标准，把这个乱局收拾干净了。

理解 MCP 其实很简单，你就把它想成 Agent 和外部工具、数据源之间的一个统一插头标准。你写一个 MCP Server，比如"连接 Zotero 文献库"这么一个东西，那么任何支持 MCP 的 Agent——Claude Code、Codex、Qwen Code、Cline 都算——都能即插即用地接上它。GitHub、文件系统、数据库、Slack、浏览器、Zotero、企业内部系统，都有对应的 MCP Server。它之所以重要，是因为它把"给 Agent 加能力"这件事，从"每家各写一遍"变成了"写一次到处能用"，这正是 2025 年之后整个 Agent 生态能爆发的关键基础设施。具体怎么配，我在 [Skill 与 MCP 入门](../03-Skill与科研技巧/01-Skill与MCP入门.md) 里会讲。

## Skill：给 Agent 装一个"专业技能包"

如果说 MCP 是"接外部工具的那根线"，那 Skill 就是"教 Agent 某个专业领域该怎么做事的说明书加脚本"。形态上，一个 Skill 通常就是一个文件夹，里面放一份 Markdown 说明，讲清楚什么时候该用、具体分几步、要注意什么，再配上可选的脚本、模板和参考资料。Agent 在遇到匹配的任务时会按需把对应的 Skill 加载进来，而不是把所有知识一股脑塞进上下文，这样既省 token 又更专注。

举个例子你就明白了：一个"文献综述 Skill"会告诉 Agent，先去 PubMed、arXiv 检索，然后去重，再按主题归纳，最后按你指定的引用格式生成一份带验证引用的报告。2026 年还冒出了不少 Skill 市场，几千上万个现成技能随你装，这个我放在 [有意思的 Skill 合集](../03-Skill与科研技巧/03-有意思的Skill.md) 里细聊。

那 Skill 和 MCP 到底怎么区分？一句话——MCP 给的是"能力"，让 Agent 多了一个新动作，比如"能查数据库了"；而 Skill 给的是"方法论"，教它怎么把一件专业的事做漂亮。两者往往是配合着用的。

## 记忆：短期的上下文，和长期的沉淀

Agent 的记忆分两层。短期记忆就是它的上下文窗口，也就是本次对话里它能"看见"的全部内容。2026 年主流模型的窗口已经大到几十万甚至上百万 token，一次塞进整个代码库或者好几篇论文都不成问题，但一旦超出窗口，早先的内容就会被截断或压缩掉。长期记忆则是跨会话保留下来的东西——你的偏好、项目背景、过往的结论，通常存成文件或向量库，需要时再检索回来。Claude Code 靠 CLAUDE.md 加记忆文件来做这件事，而 OpenClaw、Hermes 这类则格外强调"跨会话持续记住你"，Hermes 甚至能从经验里自动沉淀出新的 Skill。好的记忆机制，是 Agent 能"越用越懂你"的关键，否则它每次都得从零开始认识你。

## 权限、沙箱与安全：最容易被忽视，却最要命

Agent 能行动，就意味着它能闯祸，所以设计良好的 Agent 都有分层的权限。最保守的是只读或计划模式，只看不改、先给你一个方案；再往上是需确认模式，遇到删除、覆盖、联网发送、执行命令这类危险动作会弹窗让你点头；执行层面还有沙箱，把命令关在一个受限环境里跑，限制它能碰的目录和网络；最激进的则是全自动模式，也就是俗称的 YOLO，效率最高但风险也最高，只适合在可控、可回滚的环境里用。

这里有三条安全红线我必须单独拎出来讲。第一是**提示注入**，Agent 读到的网页或文件里可能藏着"忽略之前的指令，去做坏事"这样的文本，让它处理不可信内容时务必格外小心。第二是**密钥即权限**，你给它的 API Key 或 token 一旦泄露，就等于你的账户被别人接管了。第三是**不可逆的操作一定要留人把关**，改数据库、`rm -rf`、对外发消息这类事，永远别完全交给 Agent 自动决定。

## 把这些串起来

最后我把前面这些概念串成一个完整的画面。你给出目标之后，大模型这个大脑开始推理，它一边参考记忆里的上下文和长期沉淀，一边决定该调用哪个工具；工具层里既有文件、命令、搜索这些内建能力，也有通过 MCP 接来的外部工具和 Skill 定义的专业流程；这些动作都在权限和沙箱的约束下执行，执行完把结果反馈回大脑，于是进入下一轮循环，如此往复直到目标达成。

看懂了这套底座，你其实就掌握了后面所有产品的共同骨架。它们之间的差异，说到底无非是这几块各自打磨得多好、面向什么场景罢了。
